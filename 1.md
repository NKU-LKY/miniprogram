已对照 `接口文档.md`、小程序本地实现与 `需求说明文档.txt`，结论如下。

## 总体结论

**接口文档覆盖约 70% 的核心能力，但不足以“直接替换”当前本地存储。**  
主要问题不是“缺几个 CRUD”，而是**数据模型与社交层架构不一致**，以及**若干已实现/需求中的功能在后端没有对应接口**。

当前状态：`USE_LOCAL_BACKEND = true`，`services/api/remote/` 尚未实现，业务数据全部走 `services/local/` + `wx.setStorageSync`。

## 二、架构差异（最大障碍）

本地实现把 **Observation 当作信息流主体**，点赞/评论直接挂在 `obs_id` 上；接口文档则是：

```
Observation（观测数据） → Post（社交帖子） → Comment / PostLike
```

对接时必须二选一：

1. **改前端**：观测发布成功后自动 `POST /api/posts { obsId }`，Feed/详情/互动全部走 Post 层  
2. **改后端**：在 Observation 上增加评论/点赞接口（与现有文档不一致）

此外还存在多处字段/类型差异：

| 维度 | 本地实现 | 接口文档 |
|------|---------|---------|
| ID 类型 | 字符串 `obs_xxx` | 数字 `obsId` |
| 命名 | snake_case | camelCase |
| 物种 | 字符串 `species_name` | `speciesId` + Species 对象 |
| 地点 | 字符串 `location_name` | `locationId` + Location 对象 |
| 照片 | 单字段 `photo_url` | `photos[]` 多图 + 服务端路径 |
| 精选 | `is_featured` 布尔 + 独立 store | 无 Featured；可能用 `Post.priority` 近似 |
| 隐藏记录 | 管理员把 status 改为 `rejected` | 无独立 hidden 状态；`rejected` 语义是“审阅驳回” |

## 三、接口文档缺失、但前端已实现或需求要求的功能

### 1. 站内通知（前端已完整实现）

本地有 7 种通知类型（鉴定结果、评论、回复、记录隐藏、申诉等），个人中心有通知列表。**接口文档无任何 `/api/notifications` 模块。**

需求文档标注为“可选”，但当前 UI 已依赖此功能。

### 2. 观测隐藏申诉（前端已实现）

本地有 `Appeal` 实体及管理员审核流程（`appeal-api.ts`、`moderation` 页）。**接口文档无申诉模块。**

需求文档主要描述“封禁申诉”，前端实现的是“观测被隐藏后的申诉”，两者都缺 API。

### 3. 趣闻/故事精选（需求 UC08 + 前端已实现）

需求有独立 **Featured** 实体（标题、推荐语、有效期等）；前端用 `observation_featured` 本地列表；**接口文档无 Featured 模块**，`Post.priority` 只能部分替代，无法支持有效期、推荐语等。

### 4. 评论区开关（需求 UC10 + 前端已实现）

本地有 `comments_enabled` 字段及管理员关闭/开启逻辑。**Observation/Post/Comment 均无此字段或接口。**

### 5. 鉴定认领/释放（前端已实现）

审阅员队列有 `claimIdentification` / `releaseIdentification`（`reviewer_id`、`claimed_at` 写在观测上）。**IdentificationRequest 只有 pending/identified/rejected，无认领机制。**

### 6. 鉴定“存疑”状态

需求有 `disputed`（存疑，移出队列但保留记录）。**接口文档鉴定状态只有 pending / identified / rejected。**

### 7. 其他需求中有、文档中无的实体

- **Category**（物种分类树）
- **ReviewLog**（审核日志）
- Species 的 `scientific_name`、`category_id` 等字段（API 只有 `speciesName`、`description`）

### 8. 认证与安全

- 文档要求前端传 `openid`，**未提供 `wx.login` code 换 openid 的后端接口**（需后端自行实现，不在 REST 文档里）
- **无 JWT/Session Token**，权限靠请求体传 `userId`，安全性较弱
- 需求要求 HTTPS，文档 Base URL 为 `http://1.14.75.15`

## 六、建议

### 方案 A：以现有接口为主（改动前端）

1. 建立 `utils/request.ts`，统一处理 `{ success, data, message }` 与图片 URL 拼接  
2. 发布流程：选/创 Location → 选/创 Species → `POST /api/observations` → `POST /api/posts`  
3. Feed/详情/互动全部改为 Post 维度  
4. 精选暂用 `Post.priority` 近似实现  
5. **通知、申诉、评论区开关、鉴定认领**：要么砍掉 UI，要么请后端补接口  

### 方案 B：扩展后端（更贴合需求与现有前端）

建议后端新增或扩展：

```
POST   /api/auth/wechat          # code 换 openid + 登录
GET/POST /api/notifications      # 站内通知
POST/GET /api/appeals              # 观测隐藏申诉
GET/PATCH /api/featured            # 精选内容（或扩展 Post）
PATCH  /api/posts/:id/comments-enabled
PATCH  /api/identification-requests/:id/claim   # 认领/释放
```

并在 Observation 或 Post 上区分 **hidden（管理员隐藏）** 与 **rejected（审阅驳回）**。

## 最终答案

**当前接口文档对“观测 + 用户 + 物种 + 地点 + 鉴定”主干够用，但对现有小程序整体不够用。**  
若不做模型改造和功能取舍，至少需要：

1. 引入 **Post 中间层**（或改后端让评论/点赞直接挂 Observation）  
2. 补充 **通知、申诉、精选、评论区开关、鉴定认领** 等接口  
3. 解决 **openid 获取、HTTPS、鉴权** 等基础设施问题
